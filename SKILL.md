# Task Budget - 开放式任务预算守门员 (autoresearch 范式 + 轮次衰减)

## 为什么需要

OpenClaw agent 的 context 截断 + 并发上限**目前都没做**。任何「开放式」任务(收网型、调研型、汇总型、迭代型、对比型)会:

1. 反复调工具,工具结果塞 context
2. context 单调增长,V8 堆触顶
3. 多 agent 并发跑,内存叠加
4. **触发 OOM killer 盲杀 worker + 系统 reboot**

证据: 2026-06-24 22:07 巴拉巴西任务就是这个剧本。

**autoresearch 给了我们答案**:不要让 LLM 思考"还要不要继续搜",让它**按"衰减信号"自动终止**。

## 触发词

- "帮我收集/汇总/调研/分析/整理/梳理..."
- "看下 X 都有什么..."
- "对比 A 和 B..."
- "找一下 X 的资料..."
- "研究一下..."
- 任何**没有明确「数量 + 截止」**的任务描述

## 核心哲学(autoresearch 4 条)

1. **单一可改对象**:agent 只动一个目标(文件/数据/表),其它都是"基础设施"不改
2. **固定时间/资源预算**:每次实验预算相同,比如 60s + N 轮,不让 LLM 主观延长时间
3. **单指标 + 衰减信号**:一个标量,轮次衰减自动终止
4. **中间件不进 context**:所有原始数据写文件,LLM 只看摘要 + 决策
5. **轮数用户可配,默认 20**:派发前用户决定轮次上限,默认 20 轮覆盖 90% 场景

## ⭐ 轮次衰减协议(核心)

**这是 autoresearch 真正精髓——用"指标变化率"作为终止器,而不是让人写"什么时候停"。**

### 原理

```
轮次     本轮新增      累计    衰减率    终止判断
1        3            3       100%      继续
2        2            5       67%       继续(>30%)
3        1            6       33%       继续(>30%)
4        0            6       0%        ← 衰减触发,立即停
```

**默认轮次上限 = 20 轮**(用户可调,v0.4.0 写死 5 轮太小,实测 X 帖场景跑到第 7 轮才衰减到位)

**衰减信号比绝对值更优**:
- ✅ 用户不需要猜"3 本"还是"5 本"——衰减到位就停
- ✅ 数据稀疏场景(只找到 2 本)不会浪费 3 轮预算
- ✅ 数据密集场景(找到 50 本)也不会超调

### 三层终止条件(叠加生效)

```
【硬上限】轮次 ≤ N(用户可配,默认 20)
【软上限】衰减到位 + 连续 K 轮无新增(信号终止)
【目标上限】累计达到用户期望量(可选,优先级最高)
```

**轮次上限的常见档位**:
- **5 轮**(数据稀疏 + 高确定性,如"找 1 本具体书")
- **20 轮**(默认,大多数开放式调研/汇总)
- **50 轮**(大型扫描,如"扫 10 个产品的 X 维度")
- **100 轮**(研究级别,如"系统综述")

任一触发即停,**优先级**:目标达成 > 衰减到位 > 轮次用尽

### 协议模板

```markdown
【轮次衰减协议 - 强制】

目标产物: <文件路径 / JSON / 飞书文档>
单指标: 累计命中数(去重) / 累计覆盖度 / 累计完整度
轮次上限: ≤ N 轮(用户指定,默认 20)
单轮预算: ≤ 30s / ≤ 1 工具(v0.4.1 收紧,避免踩线)

【轮数定义 - 重要】(v0.4.1 新增)
- **搜索轮(search_round)**:每次"调工具抓新内容"算 1 轮
- **收尾轮(finalize_round)**:写入文件 + 验证不计入轮数上限
- 终止判断只看 search_round 是否达到上限

【终止条件 - 优先级递减】
1. 累计 ≥ 目标值 → 立即停(成功)
2. 连续 2 轮本轮新增 = 0 → 立即停(衰减到位)
3. 轮次达到上限 → 强制停(兜底)

【每轮强制汇报】(只对 search_round)
search_round N (耗时 Xs):
  本轮新增: <X>
  累计: <Y>
  衰减率: <本轮/上轮 或 本轮/首轮>
  决策: keep / discard / end
  next: <下一步 或 终止 - 原因>

【禁止】
- 隐瞒工具失败
- 用"我记忆里知道"伪造未抓取的内容
- 在衰减信号触发后继续搜
- finalize_round 超过 1 次(收尾只允许 1 次)
```

### 真实场景示例

**巴拉巴西书籍(密度高,1 轮停)**:
```
round 1 (12s): Open Library 搜 → 3 本 (Barabási, Easley, Watts)
  本轮新增=3, 累计=3, 衰减率=100%
  决策: keep + end(目标达成)
→ 终止,1 轮完成 ✅
```

**巴拉巴西 X 帖(密度稀疏,3 轮停)**:
```
round 1 (28s): X API 关键词 "Barabási" → 8 条
  累计=8, 决策: keep
round 2 (25s): 扩展 "network science" → 2 条
  本轮新增=2, 累计=10, 衰减率=25%
  决策: keep(>30% 不算衰减)
round 3 (30s): 关键词 "Barabási 2024" → 0 条
  本轮新增=0, 累计=10, 衰减率=0%
  决策: discard
round 4 (跳过): 连续 2 轮新增 = 0 → 触发终止
→ 终止,3 轮完成 ✅
```

**网络科学论文(中等密度,2 轮停)**:
```
round 1 (22s): arXiv API "network science" → 5 篇
  累计=5, 决策: keep
round 2 (28s): 引用追踪 + 作者扩展 → 1 篇
  本轮新增=1, 累计=6, 衰减率=20%
  决策: discard
round 3 (跳过): 连续 2 轮新增 = 0... 
  不!round 2 是 1 不是 0,衰减率 20% 但没连续 2 轮 0
  → 继续 round 3
round 3 (30s): 关键词再扩展 → 0 条
  累计=6, 决策: discard
round 4 (跳过): 连续 round 2 (新增 1) + round 3 (新增 0)... 
  
  ⚠️ "连续 2 轮新增 = 0" 要严格定义 = round N=0 且 round N-1=0
  round 2=1, round 3=0 → 不算连续 2 轮 0
  → 必须等 round 4
round 4 (30s): 完全不同关键词 → 0 条
  累计=6, 决策: discard
→ 连续 round 3 + round 4 = 0 → 触发终止,4 轮完成
```

## 切分子任务策略(多类信息场景)

**当任务范围 ≥ 3 类时**(书 + 论文 + 帖),强制切分:

```
                  ┌─ 子任务 1: 书籍
                  │   协议: 轮次衰减 (≤5 轮 / 30s 单轮)
                  │   产物: <task-dir>/books.json
[orchestrator] ──┤
                  ├─ 子任务 2: 论文
                  │   协议: 轮次衰减 (≤5 轮 / 30s 单轮)
                  │   产物: <task-dir>/papers.json
                  │
                  └─ 子任务 3: X 帖
                      协议: 轮次衰减 (≤5 轮 / 30s 单轮)
                      产物: <task-dir>/tweets.json
                      
                  ↓
              orchestrator 聚合 3 个 JSON → result.md
              (纯文件操作,不再调 web)
```

**关键设计**:
1. **子任务互相隔离**——一个失败不影响其他
2. **每个子任务内存峰值 < 200MB**(3 个并发 < 600MB)
3. **聚合阶段不调 web**,纯文件读写 → context 0 增长
4. **总时间 = max(子任务) + 聚合时间**(并行 ≤ 30s + 5s)

**子任务必须返回标准 JSON**:
```json
{
  "status": "success" | "failed",
  "rounds_run": 3,
  "total_items": 8,
  "items": [...],
  "failure_reason": "...(仅 failed 时)"
}
```

**禁止 "知识兜底"**——sub-agent 工具失败 = `status: "failed"`,**不允许塞"我记忆里的"知识进 items 数组**。

## 5 个场景模板(autoresearch + 衰减化)

### 模板 A: 信息收集(单类) ⭐ 推荐用轮次衰减

```markdown
目标: ~/openclaw-tasks/<id>/<类>.json (JSON 数组)
协议: 轮次衰减 / ≤ N 轮(默认 20) / 单轮 ≤ 30s ≤ 1 工具
指标: 累计命中数(去重)
终止: 目标达成 / 连续 2 轮新增=0 / 轮次用尽
禁止: 知识兜底 / 隐瞒工具失败
```

### 模板 B: 调研对比(A vs B vs C) ⭐ 推荐用切分

```markdown
任务: A vs B vs C 对比
切分: 
  - 子任务 A (走模板 A)
  - 子任务 B (走模板 A)
  - 子任务 C (走模板 A)
聚合: orchestrator 用本地 LLM 生成对比表(≤200 字)
```

### 模板 C: 资料汇总(多类) ⭐ 推荐用切分 + 衰减

```markdown
任务: 论文 + 帖子 + 文章
切分:
  - 论文: 轮次衰减协议 (arXiv API) / 20 轮
  - 帖子: 轮次衰减协议 (X API) / 20 轮
  - 文章: 轮次衰减协议 (博客 RSS) / 20 轮
聚合: 3 个 JSON → Markdown 表格
```

### 模板 D: 持续监控/迭代类 ⚠️ 高危

```markdown
🛑 默认拒绝(单次快照也有累积风险)

如必须:
协议: 轮次衰减 / 1 轮 / 1 工具(只跑 1 轮就停)
指标: 异常项数对比基线
⚠️ 禁用"再扫一次",衰减 = 0 时强制停
```

### 模板 E: 探索性(用户没明确范围) 🛑

```markdown
🛑 默认拒绝,反问用户:
- 你想了解 X 的什么?
- 要多少?(数字)
- 用在哪?
- 怎么判断"够好"?(单指标)
```

## 与 OpenClaw agent 配合的关键设计

**1. 任务 ID + 目录隔离**
```bash
TASK_ID=$(date +%Y%m%d-%H%M%S)-$(uuidgen | head -c 8)
mkdir -p ~/openclaw-tasks/$TASK_ID/raw
```

**2. 中间件路径(LLM 不直接读 raw)**
- LLM 拿到的工具结果会被截断到 100KB
- 超过 100KB 的内容**强制写文件** `~/openclaw-tasks/<id>/raw/<n>.md`
- LLM 只看 `ls ~/openclaw-tasks/<id>/raw/` 的输出 + 摘要

**3. 衰减信号由 sub-agent 自评 + 强制 JSON 输出**
- 派发指令里**明确要求每轮填齐 5 字段**(本轮新增 / 累计 / 衰减率 / 决策 / next)
- JSON 输出包含 `rounds_run` 和 `total_items`,方便 orchestrator 校验

**4. orchestrator(主 session)不调 web**
- 只做文件读写 + 聚合 → context 不增长
- 主 session 永远稳态

## 失败处理

**用户在群里坚持不给预算**:
- 🛑 不派 agent
- 回复: "开放式任务必爆 V8。选个模板填数字 + 单指标?"
- 等用户填数字再派

**实验跑到一半异常**:
- V8 触顶 768MB 主动 throw → OpenClaw 报 OOM → 保留 raw/ + 部分 result.json
- 用户可以拿原始文件手动汇总
- 任务记录写进 `MEMORY.md` 的 lessons

## 配套约定(已落地)

- **OpenClaw systemd NODE_OPTIONS=--max-old-space-size=768** (2026-06-24 修复)
  - V8 主动 throw OOM,比系统 OOM killer 杀 worker 优雅
- **Swap 2GB** (2026-06-12 修复)
  - 救 RSS 进程,不救 V8 堆
- **MEMORY.md 注入税** (2026-06-24 评估)
  - 实际 14KB,误报 174KB 是 OpenClaw 统计 bug
  - 不写流水账

## 实战示例

**巴拉巴西任务 - v0.4.1 派发**(默认 20 轮,用户可在群里覆盖):

```markdown
任务切分(默认 20 轮 / 30s 单轮 / 单轮 ≤ 1 工具):
├─ 子任务 1: 书籍 → sessions_spawn sub-agent
├─ 子任务 2: 论文 → sessions_spawn sub-agent
└─ 子任务 3: X 帖 → sessions_spawn sub-agent

聚合: 主 session 读 3 个 JSON → 写 result.md

预期(基于 v0.4.0 实测):
- books: 6 search_round(前 4 失败) → 1 finalize_round → 6 本
- papers: 3 search_round → 5 篇(目标达成停)
- tweets: 7 search_round → 3 条(衰减到位停)
- 总墙钟(并行): < 2 分钟
```

## 与 karpathy/autoresearch 的差异

| 维度 | autoresearch | task-budget v0.4.0 |
|---|---|---|
| 单一目标 | train.py | result.md / JSON |
| 时间预算 | 5 min | 30s 单轮 / N 轮上限(用户可配,默认 20) |
| 指标 | val_bpb | 累计命中数 |
| 终止信号 | val loss 衰减 | **新增数衰减(信号终止)** |
| 决策 | keep / discard | keep / discard / end |
| 中间件 | checkpoint | raw/ 文件夹 |
| 循环 | 无人值守过夜 | 单次任务,可多次子任务并行 |

**核心借鉴**:
1. **不让 LLM 思考"还要不要继续"**——衰减信号自动终止
2. **不让 LLM 直接读 raw 数据**——强制写文件
3. **二元决策不商量**——keep/discard/end 三态明确
4. **每轮必有汇报**——透明可中断
